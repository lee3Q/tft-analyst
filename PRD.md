# TFT Analyst — PRD (Product Requirements Document)

> 개인용 TFT 분석가 도구. Claude Code 스킬 + MCP 서버로 구현.

## 1. 개요

TFT(팀파이트 택틱스) 공략 사이트 데이터를 크롤링하여 정제하고, Claude Code 세션에서 TFT 분석가 페르소나로 게임 전/중/후 가이드를 제공하는 개인용 도구.

## 2. 핵심 결정사항

| 항목 | 결정 |
|------|------|
| 대상 사용자 | 본인 (개인용) |
| 형태 | Claude Code 스킬(`/tft`) + MCP 서버 |
| 데이터 소스 | lolchess.gg, metatft 크롤링 |
| 크롤링 방식 | 헤드리스 브라우저 + 내부 API 역분석 (사이트별 최적 선택) |
| AI 활용 | 크롤링 데이터 정제에만 LLM 사용, 서비스 작동은 규칙 기반 |
| 데이터 갱신 | 패치마다 수동 트리거 (CLI) |
| 데이터 저장 | 로컬 JSON/SQLite |
| 언어 | 한국어 전용 |
| 수익화 | 없음 (개인용) |
| 로그인 | 없음 |

## 3. 사용 플로우

### 3.1 진입
`/tft` 스킬 실행 → TFT 분석가 페르소나 활성화 → 바로 패치 브리핑 + 퀴즈 시작

### 3.2 게임 전 — 브리핑 & 퀴즈 (스킵 가능)
- 현재 패치 핵심 변경사항 브리핑
- 빌드업 지식 퀴즈: "현재 가장 유리한 빌드업은?", "가장 강력한 초반 빌드업은?" 등
- 목적: 실시간 분석가가 도와주기 어려운 **초반 빌드업 승패 판단** 지식 확인
- 퀴즈는 정제된 빌드업 데이터에서 규칙 기반 자동 생성

### 3.3 게임 중 — 실시간 코칭
- **초반 (빌드업 단계)**: 빌드업 데이터 기반 추천
  - 보유 챔피언 + 아이템 입력 → 빌드업 추천
  - 아이템별 상성 고려 (노템/구인수/대천사 등)
- **중반 이후 (강화 단계)**: 완성 덱 데이터 기반 추천
  - 플레이어가 "돈 써서 강화할 때"라고 판단한 시점부터 전환
  - 맥락으로 자동 감지 (명시적 단계 선언 불필요)
- **추천 방식**: 리스크 vs 안정 두 루트 제시
  - 안정 루트: 전환 비용 적고 평균 순위 안정적인 덱
  - 리스크 루트: 1등 확률 높은 덱
  - 순위별 복수 선택지
- **배치 추천**: ASCII 격자 아닌 텍스트 규칙
  - "상대 딜러에 맞춰 이동", "특정 챔피언 있으면 딜러 앞으로" 등 조건부 가이드
- **추가 입력**: 기본은 챔피언+아이템, 선택적으로 레벨/골드/스테이지/체력 추가 가능
- **메타 덱 조회**: 간결 모드(상위 5개) / 전체 모드(S/A/B 티어) 전환 가능

### 3.4 게임 후 — 결과 분석
- 사용자가 순위만 말하면 나머지(덱, 아이템, 실수/개선 포인트)는 대화 맥락에서 자동 추출
- 분석 방식: 규칙 기반 누적 통계 + LLM 세션 맥락 참고 복기
- `record-game`으로 자동 저장

### 3.5 세션 단위
- 1세션 = 1게임
- `/tft` 진입 → 퀴즈 → 코칭 → 분석/기록 → 종료
- 다음 게임은 `/tft` 재실행

## 4. MCP 도구 (7개)

| 도구 | 기능 |
|------|------|
| `crawl` | 크롤링 실행 (CLI 트리거) |
| `get-meta` | 메타 덱 조회 (간결/전체 모드) |
| `recommend` | 상황 기반 종합 추천 (빌드업/완성덱, 리스크/안정, 상성/배치/아이템) |
| `record-game` | 게임 결과 기록 (세션 대화에서 자동 추출) |
| `get-stats` | 누적 통계 조회 (덱별 평균 등수, 승률 등) |
| `quiz` | 패치 브리핑 + 빌드업 퀴즈 |
| `add-buildup` | 빌드업 데이터 등록/관리 |

## 5. 데이터 스키마

### 5.0 챔피언 (Set 16 기준)
```json
{
  "name": "챔피언 이름",
  "cost": 3,
  "synergies": ["시너지 태그"],
  "skill_name": "스킬 이름",
  "skill_description": "스킬 효과 요약",
  "skill_range": 3,
  "skill_target": "farthest|nearest|lowest_hp|highest_hp|aoe|self|random|line",
  "cc_type": "stun|knockup|pull|slow|silence|taunt|disarm",
  "cc_duration": 1.5,
  "damage_type": "physical|magic|true",
  "attack_range": 1,
  "role": "tank|carry|support|assassin"
}
```
> 배치 가이드는 이 데이터 기반으로 동적 생성. 하드코딩 규칙 없음.

### 5.1 완성 덱
```json
{
  "name": "덱 이름",
  "core_champions": ["챔피언 목록"],
  "recommended_items": {"챔피언": ["아이템"]},
  "avg_placement": 3.2,
  "first_rate": 0.15,
  "difficulty": "중",
  "synergy_tags": ["시너지 태그"],
  "recommended_augments": ["증강 목록"],
  "emblem_synergies": ["상징 시너지"]
}
```

### 5.2 빌드업
```json
{
  "name": "빌드업 이름",
  "champions": ["초반 챔피언 조합"],
  "tier": "S",
  "matchups": {
    "strong_against": ["유리한 빌드업"],
    "weak_against": ["불리한 빌드업"]
  },
  "item_matchups": {
    "no_item": "상성 설명",
    "recurve_bow": "상성 설명",
    "archangels": "상성 설명"
  },
  "transition_decks": ["전환 가능 완성 덱"]
}
```

## 6. 기술 아키텍처

```
/tft 스킬 (페르소나 + 플로우 관리)
    ↕
Claude Code 하네스
    ↕
MCP 서버 (7 tools)
    ↕
로컬 데이터 (JSON/SQLite)
    ↑
크롤러 (Playwright/내부 API) → LLM 정제
```

## 7. 데이터 소스 주의사항

- lolchess.gg, metatft 모두 SPA — 정적 크롤링 불가
- 헤드리스 브라우저 또는 내부 API 역분석 필요
- 초반 빌드업 데이터는 크롤링 소스 미확인 — 고수 유튜브 참고 + 실제 플레이 데이터 수동 입력으로 구축
- 패치마다 수동 트리거로 갱신

## 8. 추천 로직

- **최적 기준**: 승률 최우선
- **충돌 처리**: AI(LLM)가 여러 소스를 종합 판단 (크롤링 정제 단계에서 처리)
- **초반/중반 전환**: 사용자 맥락으로 자동 감지 (플레이어가 강화 시점 판단)
- **출력**: 간결/상세 모드 사용자 전환 가능
